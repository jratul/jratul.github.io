# OSI 7계층과 TCP/IP

네트워크를 공부할 때 가장 먼저 만나는 개념이다. 처음에는 왜 이걸 배우는지 몰라서 그냥 외우려고 하는데, 나중에 실무에서 "왜 연결이 안 되지?" 디버깅할 때 계층 개념이 없으면 어디서 문제가 생겼는지 파악조차 못한다.

---

## OSI 7계층이란?

국제표준화기구(ISO)가 만든 네트워크 통신의 표준 모델이다. 복잡한 네트워크 통신을 7개 계층으로 나눠서 각 계층이 무슨 역할을 하는지 명확히 정의했다.

**왜 계층으로 나눴을까?** 음식점에 비유하면 이해하기 쉽다. 주문받는 사람, 요리사, 재료 관리자, 배달원이 각자 역할이 있다. 주문받는 사람은 요리 방법을 몰라도 되고, 요리사는 배달 경로를 몰라도 된다. 네트워크도 마찬가지다. HTTP 개발자가 케이블 신호 처리까지 알 필요 없고, 케이블 제조사가 HTTP 규격을 알 필요 없다.

```
7. 응용 계층 (Application)   — HTTP, FTP, DNS, SMTP, SSH
   사용자와 직접 맞닿는 계층. 웹 브라우저, 이메일 클라이언트.

6. 표현 계층 (Presentation) — SSL/TLS, 암호화, 압축, 인코딩
   데이터를 어떻게 표현할지. 암호화/복호화, 압축/해제.

5. 세션 계층 (Session)      — 연결 관리, 동기화
   두 시스템 사이의 대화(세션)를 관리.

4. 전송 계층 (Transport)    — TCP, UDP, 포트 번호
   데이터를 신뢰성 있게(TCP) 또는 빠르게(UDP) 전송.

3. 네트워크 계층 (Network)  — IP, 라우팅
   IP 주소 기반으로 최적 경로 선택. 택배 경로를 결정하는 것과 같다.

2. 데이터링크 계층 (Data Link) — MAC 주소, 이더넷, 스위치
   같은 네트워크 안에서의 통신. 집 안에서 공유기까지.

1. 물리 계층 (Physical)     — 케이블, 신호, WiFi
   실제 데이터를 전기 신호나 빛으로 전송.
```

**실무에서는 4계층(TCP/UDP)과 7계층(HTTP)을 가장 많이 다룬다.**

## TCP/IP 4계층 — 실제 인터넷이 사용하는 모델

OSI 7계층은 이론적인 모델이고, 실제 인터넷은 더 단순한 TCP/IP 4계층을 사용한다. OSI를 실용적으로 합친 버전이라고 보면 된다.

```
4. 응용 계층 (Application) — HTTP, HTTPS, DNS, FTP, SSH, SMTP
   OSI의 5, 6, 7계층을 합쳐서 하나로.

3. 전송 계층 (Transport)   — TCP, UDP
   OSI의 4계층과 동일. 포트 번호로 애플리케이션 구분.

2. 인터넷 계층 (Internet)  — IP, ICMP
   OSI의 3계층과 동일. IP 주소로 라우팅.

1. 네트워크 접근 계층      — 이더넷, Wi-Fi, MAC 주소
   OSI의 1, 2계층을 합쳐서 하나로.
```

## IP 주소 — 인터넷의 집 주소

IP 주소는 인터넷에서 각 기기를 식별하는 고유 번호다. 현실에서 집 주소가 있어야 우편물을 보낼 수 있는 것처럼, 인터넷에서도 IP 주소가 있어야 데이터를 보낼 수 있다.

```
IPv4: 32비트, 4개의 숫자(옥텟)로 표현
예: 192.168.1.100

각 숫자는 0~255 범위
4개 × 8비트 = 32비트
최대 주소 수: 약 43억 개

공인 IP (Public IP):
— 인터넷에서 유일한 주소
— 전 세계에서 접근 가능
— 예: 구글 서버의 IP, 내 공유기의 외부 IP

사설 IP (Private IP):
— 내부 네트워크에서만 사용
— 인터넷에서 직접 라우팅 불가 (아파트 호수 같은 것)
— NAT를 통해 공인 IP로 변환되어 외부 통신

사설 IP 범위:
10.0.0.0    ~ 10.255.255.255   (10.x.x.x)
172.16.0.0  ~ 172.31.255.255   (172.16~31.x.x)
192.168.0.0 ~ 192.168.255.255  (192.168.x.x ← 집 공유기)

특수 IP:
127.0.0.1        — loopback (자기 자신에게 보내는 것)
0.0.0.0          — 모든 인터페이스 (서버가 모든 IP에서 수신)
255.255.255.255  — 브로드캐스트 (같은 네트워크 전체에 발송)
```

## 서브넷 마스크 (CIDR) — 네트워크 범위 지정

`192.168.1.0/24`처럼 IP 뒤에 붙는 숫자가 CIDR 표기법이다. `/24`는 앞에서 24비트가 네트워크 주소라는 뜻이다.

**아파트 주소로 비유하면:**
- `192.168.1` → 아파트 단지 번호 (네트워크 주소)
- `.100` → 몇 호 (호스트 주소)

```
192.168.1.0/24
          ↑
          24비트가 네트워크 주소, 나머지 8비트가 호스트 주소

/32 → 단일 호스트 1개    (255.255.255.255)
/24 → 호스트 256개       (255.255.255.0)   ← 가장 흔함
/16 → 호스트 65,536개    (255.255.0.0)
/8  → 호스트 1,677만 개  (255.0.0.0)

주의: 각 서브넷에서 첫 번째 주소(네트워크 주소)와
      마지막 주소(브로드캐스트)는 실제 사용 불가
      → /24면 실제 사용 가능한 것은 254개

AWS VPC 예시 (실무에서 자주 쓰는 구조):
VPC CIDR:       10.0.0.0/16       → 65,536개 IP
Public Subnet:  10.0.1.0/24       → 256개 IP (외부 접근 허용)
Private Subnet: 10.0.2.0/24       → 256개 IP (내부 전용)
DB Subnet:      10.0.20.0/24      → 256개 IP (DB 전용, 격리)
```

## TCP — 등기우편 같은 프로토콜

TCP는 **신뢰성**이 핵심이다. 데이터가 반드시 목적지에 도착하는지, 순서가 맞는지 보장한다. 등기우편처럼 "잘 받았어요" 확인이 있다.

```
3-way Handshake (연결 수립):
클라이언트 → 서버: SYN         "연결해도 될까요?"
서버 → 클라이언트: SYN-ACK     "네, 좋아요!"
클라이언트 → 서버: ACK         "알겠어요, 시작합시다!"
→ 연결 수립 완료. 이제 데이터 전송 가능.

SYN  — Synchronize: 연결 시작 요청
ACK  — Acknowledge: 수신 확인
seq  — Sequence Number: 패킷 순서 번호
ack  — Acknowledgment Number: 다음에 받을 seq 번호

4-way Handshake (연결 종료):
클라이언트 → 서버: FIN         "저는 다 보냈어요"
서버 → 클라이언트: ACK         "알겠어요 (서버는 아직 보낼 것 있을 수 있음)"
서버 → 클라이언트: FIN         "저도 다 보냈어요"
클라이언트 → 서버: ACK         "알겠어요, 끊을게요"
→ 연결 종료
```

```
TCP 특성:
✅ 순서 보장  — 패킷이 다른 순서로 도착해도 올바른 순서로 재조립
✅ 신뢰성    — 손실된 패킷 자동 재전송
✅ 흐름 제어  — 수신자가 처리할 수 있는 속도로 전송 조절
✅ 혼잡 제어  — 네트워크가 혼잡하면 전송 속도 자동 감소
❌ 오버헤드   — 연결 수립/유지 비용이 있어 UDP보다 느림

주요 사용처: HTTP, HTTPS, SSH, MySQL, PostgreSQL, 이메일
```

## UDP — 일반우편 같은 프로토콜

UDP는 **속도**가 핵심이다. "잘 받았어요" 확인 없이 그냥 보낸다. 중간에 잃어버려도 재전송하지 않는다. 대신 매우 빠르다.

```
UDP 특성:
✅ 빠름       — 연결 수립 과정 없이 바로 전송
✅ 낮은 지연   — 실시간 통신에 적합
✅ 가벼운 헤더 — 8바이트 (TCP는 20바이트 이상)
❌ 순서 보장 없음 — 먼저 보낸 것이 나중에 도착할 수 있음
❌ 재전송 없음    — 손실된 패킷 그냥 버림
❌ 흐름/혼잡 제어 없음

주요 사용처:
— DNS     : 빠른 도메인 조회가 중요. 실패하면 다시 요청.
— 온라인 게임 : 약간의 패킷 손실 허용, 지연 최소화가 중요
— 동영상 스트리밍 : 잠깐 화질 나빠져도 중단은 더 곤란
— WebRTC  : 화상통화, 실시간 P2P 통신
— DHCP    : IP 주소 자동 할당 (브로드캐스트 필요)
```

## 포트 번호 — 방 번호처럼 서비스를 구분

IP 주소가 건물 주소라면, 포트 번호는 방 번호다. 같은 서버에서 웹(80), SSH(22), DB(5432)가 동시에 돌아갈 때 포트 번호로 구분한다.

```
포트 범위:
0 ~ 1023      — Well-known 포트 (루트 권한 필요, 표준 서비스)
1024 ~ 49151  — 등록된 포트 (일반 앱 서버)
49152 ~ 65535 — 동적/사설 포트 (클라이언트가 임시로 사용)

꼭 알아야 할 포트:
22    — SSH (원격 접속)
25    — SMTP (이메일 발송)
53    — DNS (도메인 조회)
80    — HTTP
443   — HTTPS (TLS 암호화)
3306  — MySQL
5432  — PostgreSQL
6379  — Redis
8080  — HTTP 대체 (개발/앱 서버에서 자주 씀)
27017 — MongoDB
```

## 패킷의 여정 — 계층별 데이터 흐름

브라우저에서 `https://api.example.com/users`를 요청하면 어떤 일이 벌어지는지 단계별로 살펴보자.

```
1. DNS 조회: api.example.com → 93.184.216.34 (IP 주소 알아내기)
2. TCP 연결: 3-way handshake (포트 443)
3. TLS 핸드셰이크: 암호화 협상 (HTTPS)
4. HTTP 요청: GET /users HTTP/1.1 전송
5. 서버 처리: DB 조회, 비즈니스 로직 실행
6. HTTP 응답: 200 OK + JSON 바디
7. TCP 연결 종료 또는 재사용 (Keep-Alive)

보내는 쪽 (계층 내려가며 헤더 추가):
7계층: HTTP 요청 생성
4계층: TCP 헤더 추가 (출발지/목적지 포트)
3계층: IP 헤더 추가 (출발지/목적지 IP)
2계층: MAC 주소 추가
1계층: 전기 신호로 변환 → 전송

받는 쪽 (계층 올라가며 헤더 제거):
1계층: 신호 수신
2계층: MAC 확인, 헤더 제거
3계층: IP 확인, 헤더 제거
4계층: 포트 확인, 재조립
7계층: HTTP 요청 파싱 → 처리
```

## 실무에서 계층 개념을 사용하는 순간

```
L4 로드밸런서 vs L7 로드밸런서:

L4 (TCP 레벨에서 분산):
— IP:Port 기준으로만 라우팅
— HTTP 내용은 보지 않음
— 빠르고 단순
— AWS NLB (Network Load Balancer)

L7 (HTTP 레벨에서 분산):
— URL 경로, 헤더, 쿠키 기준 라우팅
— /api/* → 앱 서버 그룹
— /static/* → 파일 서버 그룹
— SSL 종료, 압축, 캐싱 가능
— AWS ALB (Application Load Balancer), Nginx

Kubernetes:
Service (ClusterIP/NodePort) → L4 로드밸런싱
Ingress                      → L7 로드밸런싱
```

## "연결이 안 된다"를 계층별로 디버깅하는 법

문제가 생겼을 때 1계층부터 올라가면서 확인하면 빠르게 원인을 찾을 수 있다.

```bash
# 1단계: 물리/IP 연결 확인 (L1~L3)
ping 10.0.1.10                          # 상대방 IP까지 도달하나?
traceroute 10.0.1.10                    # 어디서 패킷이 막히나?

# 2단계: 포트 연결 확인 (L4)
ss -tlnp | grep 8080                    # 서버에서 해당 포트 열려있나?
nc -zv 10.0.1.10 8080                   # 외부에서 접근 가능한가?

# 3단계: 방화벽/보안그룹 확인
sudo ufw status                         # 리눅스 방화벽 상태
# AWS: 보안 그룹 인바운드 규칙 확인

# 4단계: DNS 확인 (L7)
dig api.example.com                     # 도메인이 올바른 IP로 해석되나?
nslookup api.example.com

# 5단계: 앱 프로세스 확인
systemctl status myapp                  # 서비스 실행 중인가?
journalctl -u myapp -n 50               # 최근 로그에 에러 있나?
```

## 초보자가 자주 하는 실수

```
실수 1: "포트가 열려있으면 당연히 연결되겠지"
→ 포트가 열려있어도 방화벽(보안 그룹)이 막고 있을 수 있다.
→ AWS에서 EC2 포트 8080이 열려도 보안 그룹에서 막으면 접근 불가.

실수 2: "같은 서버인데 왜 외부에서 연결이 안 되지?"
→ 서비스가 127.0.0.1(localhost)에만 바인딩되어 있을 수 있다.
→ ss -tlnp 실행 시 127.0.0.1:8080 이면 외부 접근 불가.
→ 0.0.0.0:8080 이어야 외부에서 접근 가능.

실수 3: "개발에서는 됐는데 운영에서 안 된다"
→ 사설 IP와 공인 IP를 혼동. VPC 안에서는 사설 IP로 통신.
→ 방화벽 규칙이 개발/운영 환경마다 다름.

실수 4: "TIME_WAIT 상태의 소켓이 너무 많다"
→ HTTP/1.0이나 Keep-Alive 없는 연결에서 자주 발생.
→ 연결마다 TCP 연결/종료를 반복하면 TIME_WAIT가 쌓임.
→ Connection: keep-alive 설정으로 연결 재사용.
```

## TCP 소켓 상태 — 연결의 생애 주기

```
LISTEN      — 서버가 연결 대기 중 (포트 열려있음)
SYN_SENT    — 클라이언트가 SYN 보낸 상태
SYN_RECV    — 서버가 SYN 받고 SYN-ACK 보낸 상태
ESTABLISHED — 연결 완료, 데이터 전송 중
FIN_WAIT_1  — FIN 보낸 상태 (종료 시작)
FIN_WAIT_2  — 상대방 FIN 기다리는 상태
TIME_WAIT   — 마지막 ACK 후 대기 (2MSL, 약 60초)
CLOSE_WAIT  — 상대방 FIN 받고 로컬 FIN 대기 ← 버그 신호!
LAST_ACK    — FIN 보내고 ACK 기다리는 상태
CLOSED      — 완전히 종료

TIME_WAIT가 많으면: 짧은 수명의 HTTP 연결이 많다는 뜻. 정상이지만 포트 고갈 가능.
CLOSE_WAIT가 많으면: 앱에서 연결을 제대로 닫지 않는 버그! 즉시 수정 필요.
```

```bash
# 소켓 상태 확인
ss -tn                                  # 모든 TCP 연결 보기
ss -tn | awk '{print $1}' | sort | uniq -c  # 상태별 개수 세기
```
