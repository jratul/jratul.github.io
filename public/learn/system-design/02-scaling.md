# 스케일링 전략

트래픽 증가에 대응하는 방법.

---

## 수직 확장 vs 수평 확장

```
수직 확장 (Scale Up):
— 서버 스펙 업그레이드 (CPU, 메모리)
— 장점: 단순, 데이터 일관성 유지
— 단점: 한계 있음, SPOF, 비용 증가
— 언제: 빠른 임시 해결, DB 서버

수평 확장 (Scale Out):
— 서버 대수 증가
— 장점: 이론상 무한 확장, 고가용성
— 단점: 복잡성 증가, 상태 관리 어려움
— 언제: 웹 서버, 마이크로서비스

현실적 접근:
1. 수직 확장으로 시작 (단순)
2. 한계 도달 시 수평 확장
3. 서비스 분리 (마이크로서비스)
```

## 계층별 스케일링

```
웹 서버 계층:
— 상태 없음 (Stateless) 설계
— 세션 → Redis에 저장
— 로드 밸런서로 분산
— 오토 스케일링 적용

애플리케이션 계층:
— 서비스별 독립 스케일링
— 무거운 작업 → 비동기 큐
— 읽기 전용 서비스 분리

데이터 계층:
— 읽기: Read Replica 추가
— 쓰기: 샤딩 (Sharding)
— 캐시 계층 추가 (Redis)

캐시 계층:
— CDN: 정적 파일
— 애플리케이션 캐시: 자주 쓰는 데이터
— DB 캐시: 쿼리 결과
```

---

## 데이터베이스 스케일링

```
읽기 복제 (Read Replica):
Primary  ← 쓰기
   ↓
Replica1 ← 읽기
Replica2 ← 읽기
Replica3 ← 읽기

적용 시점: 읽기 > 쓰기 비율이 압도적일 때

파티셔닝:
수직 파티셔닝 (Column Partitioning):
— users 테이블 → user_basic + user_profile
— 자주 쓰는 컬럼만 빠른 테이블에

수평 파티셔닝 (Row Partitioning = Sharding):
— user_id % 4 → shard 0, 1, 2, 3
— 각 샤드가 독립 DB 서버

Federation (기능별 분리):
— users DB, products DB, orders DB
— 서비스별 독립 DB
```

## 샤딩 (Sharding)

```
샤드 키 선택 (핵심!):

좋은 샤드 키:
— 데이터가 균등하게 분산
— 핫스팟 없음
— 대부분의 쿼리에 포함

나쁜 샤드 키:
— 단조 증가 ID → 마지막 샤드만 집중
— 시간 → 최근 샤드에 집중

샤딩 방법:

Hash Sharding:
shard = hash(user_id) % shard_count
장점: 균등 분산
단점: 리샤딩 시 대부분 재이동

Range Sharding:
shard 0: user_id 0-999999
shard 1: user_id 1000000-1999999
장점: 범위 쿼리 효율적
단점: 핫스팟 가능

Consistent Hashing:
— 링 형태로 노드 배치
— 노드 추가/제거 시 최소 재분배
— 가상 노드로 균등 분산 보장
```

---

## Consistent Hashing

```
원리:
1. 해시 공간을 0~2^32 링으로
2. 서버와 데이터를 링에 배치
3. 데이터 → 시계 방향으로 가장 가까운 서버

서버 A: hash("ServerA") = 1000
서버 B: hash("ServerB") = 3000
서버 C: hash("ServerC") = 6000

데이터 X: hash("X") = 2000 → 서버 B
데이터 Y: hash("Y") = 4000 → 서버 C

서버 추가 시:
— 새 서버 D(hash=5000) 추가
— 데이터 Y(4000)만 D로 이동
— 나머지 데이터 이동 없음

가상 노드 (Virtual Nodes):
— 서버 A → A1, A2, A3 여러 위치
— 더 균등한 분산
— 서버 용량에 비례하여 가상 노드 수 조정
```

## 상태 관리

```
Stateless 설계 (권장):
— 서버가 상태를 저장하지 않음
— 요청마다 완전한 정보 포함
— 어떤 서버도 처리 가능

Sticky Session (비권장):
— 같은 사용자 → 같은 서버
— 서버 장애 시 세션 손실
— 수평 확장 어려움

외부 세션 저장소:
— Redis 세션 저장
— DB 세션 저장
— JWT (토큰에 정보 포함)

JWT 장점:
— 서버리스/무상태
— 서비스 간 인증 공유
JWT 단점:
— 토큰 크기 (매 요청 포함)
— 무효화 어려움 (만료 전)
```

## 오토 스케일링

```
AWS Auto Scaling 정책:

Target Tracking:
— CPU 70% 유지
— 자동으로 인스턴스 수 조정

Step Scaling:
— CPU 60-70%: +1 인스턴스
— CPU 70-80%: +2 인스턴스
— CPU > 80%: +3 인스턴스

Scheduled Scaling:
— 매일 09:00 +5 인스턴스 (업무 시작)
— 매일 18:00 -5 인스턴스 (업무 종료)

쿨다운 기간:
— 스케일 아웃 후 300초 대기
— 불안정한 반복 스케일링 방지

고려사항:
— 워밍업 시간 (Spring Boot ~30초)
— 최소 인스턴스 유지 (0 → 1 급증 방지)
— ALB Health Check 연동
```
