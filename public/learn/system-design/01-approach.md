# 시스템 설계 접근법

---

## 시스템 설계란?

백엔드 개발자라면 "이 기능을 어떻게 구현할까?" 뿐 아니라 "이 서비스가 수백만 명의 사용자를 어떻게 처리할까?"를 고민해야 합니다. 시스템 설계는 그 질문에 답하는 과정입니다.

**비유**: 편의점을 하나 열 때는 계산대 하나면 충분합니다. 하지만 전국에 체인점을 열 때는 물류, 재고 관리, 직원 교육 체계가 필요하죠. 소프트웨어도 마찬가지입니다.

---

## 설계 면접 5단계 프레임워크

```
1단계: 요구사항 명확화 (5분)
   기능적 요구사항: 무엇을 해야 하는가?
     - URL 단축 서비스: 짧은 URL 생성, 리다이렉트
   비기능적 요구사항: 어떻게 해야 하는가?
     - DAU, QPS, 응답 시간, 가용성 목표

2단계: 규모 추정 (3분)
   - QPS (초당 요청 수), 저장소 용량, 대역폭
   - 실제 숫자를 추정해야 설계 방향이 결정됨

3단계: 고수준 설계 (10분)
   - API 엔드포인트 정의
   - 주요 컴포넌트와 데이터 흐름
   - ASCII 다이어그램으로 시각화

4단계: 상세 설계 (15분)
   - 핵심 컴포넌트 deep dive
   - 병목 지점과 해결 방법
   - 트레이드오프 명시

5단계: 마무리 (5분)
   - 장애 시나리오 (단일 장애점은?)
   - 모니터링/알람
   - 추가 개선 방향
```

## 규모 추정 (Back-of-Envelope)

면접에서 숫자를 추정하는 능력이 중요합니다. 외워야 할 기준 수치들:

```
시간 단위:
  1분 = 60초
  1시간 = 3,600초
  1일 = 86,400초 ≈ 10만 초
  1개월 ≈ 250만 초
  1년 ≈ 3천만 초

데이터 크기:
  ASCII 문자: 1 Byte
  한글 문자: 2~3 Bytes (UTF-8)
  1KB = 10^3 Bytes
  1MB = 10^6 Bytes
  1GB = 10^9 Bytes
  1TB = 10^12 Bytes

지연시간 (빠른 것부터):
  L1 캐시: 1ns
  L2 캐시: 4ns
  RAM:     100ns
  SSD:     100µs (마이크로초)
  HDD:     10ms
  LAN:     1ms
  WAN:     100ms
```

### 실제 추정 예시: 트위터

```
전제:
  DAU: 3억 명
  1인당 일평균 트윗: 2개
  읽기:쓰기 = 100:1

쓰기 QPS:
  3억 × 2 / 86,400 ≈ 7,000 QPS
  피크: 7,000 × 3 = 21,000 QPS

읽기 QPS:
  7,000 × 100 = 700,000 QPS

저장소:
  트윗 1개: 280자 × 2B ≈ 600B ≈ 1KB
  하루: 7,000 × 86,400 ≈ 6억 개
  하루 저장: 6억 × 1KB = 600GB
  5년: 600GB × 365 × 5 ≈ 1.1PB
```

## 핵심 설계 원칙

```
단일 책임 (Single Responsibility):
  각 서비스/컴포넌트는 한 가지 일만 한다
  → 변경 이유가 하나여야 한다

느슨한 결합 (Loose Coupling):
  컴포넌트 간 의존성 최소화
  → 한 쪽이 바뀌어도 다른 쪽에 영향 최소화

실패 대비 설계 (Design for Failure):
  "무엇이 잘못될 수 있는가?" 항상 질문
  서버 한 대가 죽어도 서비스가 동작해야 한다
  → Circuit Breaker, Retry, Timeout, Fallback

수평 확장 우선:
  서버 성능을 높이는 것(수직)보다
  서버 대수를 늘리는 것(수평)이 더 확장성 있음
  → 서버가 상태를 저장하지 않도록 (Stateless)

점진적 확장:
  처음부터 1억 명 대비 설계 금지
  단순하게 시작 → 병목이 생기면 최적화
```

## 핵심 트레이드오프

### CAP 정리

분산 시스템에서는 일관성(Consistency), 가용성(Availability), 분단 허용성(Partition Tolerance) 중 **두 가지만** 보장할 수 있습니다.

```
CP 시스템 (일관성 + 분단 허용):
  네트워크 분단 시 일부 요청을 거부 (가용성 희생)
  사용 사례: 은행 계좌, 재고 관리 (정확도 중요)
  예: HBase, Zookeeper

AP 시스템 (가용성 + 분단 허용):
  네트워크 분단 시에도 응답 (일관성 희생)
  잠시 오래된 데이터를 줄 수 있음 (최종 일관성)
  사용 사례: SNS 좋아요, 조회수 (정확도보다 가용성)
  예: Cassandra, DynamoDB
```

### 동기 vs 비동기

```
동기 처리:
  클라이언트가 응답을 기다림
  장점: 단순, 에러 처리 쉬움, 즉시 결과 확인
  단점: 처리가 느리면 전체가 느려짐
  사용: 결제, 로그인, 조회

비동기 처리:
  클라이언트가 "접수됐습니다"만 받고 나중에 완료
  장점: 빠른 응답, 높은 처리량, 느슨한 결합
  단점: 결과를 즉시 모름, 실패 처리 복잡
  사용: 이메일 발송, 이미지 처리, 알림
```

## 고가용성 (High Availability) 패턴

```
목표: 서비스가 99.9% 이상 시간 동안 동작

가용성 수치:
  99%    = 연 3.65일 다운 (two nines)
  99.9%  = 연 8.7시간 다운 (three nines)
  99.99% = 연 52분 다운 (four nines)
  99.999% = 연 5분 다운 (five nines)
```

고가용성을 달성하는 방법:

```
복제 (Replication):
  같은 데이터를 여러 서버에 복사
  한 서버가 죽어도 다른 서버가 서비스 지속

로드 밸런서:
  여러 서버에 요청을 분산
  비정상 서버를 자동으로 제외

헬스 체크:
  주기적으로 서버 상태 확인
  비정상 → 자동 제외 → 복구 후 재투입

Graceful Degradation:
  일부 기능이 장애나도 핵심 기능은 동작
  예: 추천 서비스 장애 → 기본 피드만 표시

Failover:
  Active-Active: 두 서버 동시 서비스 (효율적)
  Active-Passive: 하나는 대기 (안전하지만 낭비)
```

## 분산 시스템 핵심 문제

```
멱등성 (Idempotency):
  같은 요청을 여러 번 보내도 결과가 동일해야 함
  네트워크 재시도 시 중복 처리 방지
  예: 결제 요청 → 고유 idempotency_key로 중복 체크

클락 동기화:
  서버마다 시간이 약간 다름 (수 ms 오차)
  순서 보장이 필요하면 Lamport Clock 또는 Vector Clock 사용

최종 일관성 (Eventual Consistency):
  분산 시스템에서 즉시 일관성은 비용이 큼
  잠시 불일치 허용 → 나중에 일치
  예: DNS 업데이트가 전파되는 데 시간이 걸림

분산 트랜잭션:
  여러 서비스를 하나의 트랜잭션으로 묶기 어려움
  해결: Saga 패턴 (보상 트랜잭션), Outbox 패턴
```

## 실전 설계 시 자주 쓰는 컴포넌트

```
CDN (Content Delivery Network):
  전 세계 엣지 서버에 정적 콘텐츠 캐시
  사용자와 가까운 서버에서 제공 → 지연 감소

로드 밸런서:
  여러 앱 서버에 요청 분산
  L4 (TCP): 빠르지만 단순
  L7 (HTTP): 내용 기반 라우팅 가능

캐시 (Redis):
  자주 조회하는 데이터를 메모리에 저장
  DB 부하 감소, 응답 속도 향상

메시지 큐 (Kafka, SQS):
  서비스 간 비동기 통신
  느슨한 결합, 트래픽 버퍼링

DB 복제:
  읽기 부하 분산 (Primary → Replica)
  장애 시 Failover

객체 스토리지 (S3):
  이미지, 동영상 등 대용량 파일 저장
  무한 확장, 저렴한 비용
```
