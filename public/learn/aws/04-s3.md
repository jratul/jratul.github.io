# S3 (Simple Storage Service)

S3는 **무한 용량의 클라우드 하드디스크**다. 사진, 동영상, 문서, 소프트웨어 배포 파일 등 모든 종류의 파일을 저장할 수 있다. 용량 걱정 없이 쓴 만큼만 돈을 낸다.

**S3 = 클라우드 무한 저장소** — 직접 서버를 운영하지 않고 AWS가 모든 관리를 해준다.

```
S3를 쓰는 대표적인 경우:
- 이미지, 동영상 파일 저장 (사용자 업로드 파일)
- 정적 웹사이트 호스팅 (HTML, CSS, JS)
- 로그 파일 장기 보관
- DB 백업 파일 저장
- 소프트웨어 배포 파일 (CI/CD에서 JAR 파일 저장)
- 데이터 분석용 원본 데이터 저장 (데이터 레이크)
```

---

## S3 기본 개념

```
버킷 (Bucket):
- 파일을 담는 컨테이너. 폴더와 비슷한 개념.
- 이름은 전 세계에서 유일해야 함 (모든 AWS 고객 포함)
- 특정 리전에 생성됨
- 예: my-company-uploads-2024

객체 (Object):
- 실제 저장되는 파일 + 메타데이터
- 최대 파일 크기: 5TB
- Key = 파일의 경로 (예: images/2024/01/photo.jpg)
  → 실제 폴더가 아니라 Key에 / 를 써서 폴더처럼 보이게 함

주요 특징:
- 내구성: 99.999999999% (11 9's) → 1000억 개 파일 저장 시 연간 1개 손실
- 가용성: 99.99%
- 버전 관리: 같은 파일 여러 버전 보관 가능
- 수명 주기: 오래된 파일 자동 삭제 또는 저렴한 스토리지로 이동
- 이벤트: 파일 업로드 시 Lambda 자동 실행 가능
```

## S3 기본 사용법

```bash
# 버킷 생성 (이름은 전 세계 유일해야 함!)
aws s3 mb s3://my-unique-bucket-2024 --region ap-northeast-2

# 파일 업로드
aws s3 cp file.txt s3://my-bucket/            # 루트에 업로드
aws s3 cp file.txt s3://my-bucket/docs/       # 폴더(Key 경로)에 업로드
aws s3 cp file.txt s3://my-bucket/docs/renamed.txt  # 이름 변경하여 업로드

# 로컬 폴더 전체를 S3로 동기화
aws s3 sync ./local-folder s3://my-bucket/remote-folder/
# 변경된 파일만 업로드, 삭제는 자동으로 안 함
aws s3 sync ./local-folder s3://my-bucket/ --delete  # S3에서도 삭제

# 파일 다운로드
aws s3 cp s3://my-bucket/file.txt ./          # 현재 디렉토리로
aws s3 sync s3://my-bucket/remote/ ./local/   # 동기화

# 파일 목록 조회
aws s3 ls s3://my-bucket/
aws s3 ls s3://my-bucket/ --recursive          # 모든 하위 파일까지
aws s3 ls s3://my-bucket/ --recursive --human-readable --summarize  # 크기 포함

# 파일 삭제
aws s3 rm s3://my-bucket/file.txt
aws s3 rm s3://my-bucket/ --recursive          # 버킷 전체 삭제

# 버킷 삭제 (비어있어야 함)
aws s3 rb s3://my-bucket
aws s3 rb s3://my-bucket --force               # 내용 포함 강제 삭제
```

## 스토리지 클래스 — 비용 최적화

파일 접근 빈도에 따라 저장 방식을 선택하면 비용을 크게 줄일 수 있다.

```
Standard (표준):
- 자주 접근하는 파일
- 가용성 99.99%
- 저장 비용: $0.023/GB/월
- 조회 비용: 매우 저렴
- 기본 선택

Standard-IA (Infrequent Access, 비자주 접근):
- 월 1회 미만 접근하는 파일
- 저장 비용 저렴 ($0.0125/GB/월)
- 하지만 조회 시 추가 비용 발생
- 최소 30일 보관
- 예: 30일 지난 로그 파일

One Zone-IA:
- 단일 AZ에만 저장 (가용성 낮음)
- IA보다 20% 저렴
- AZ 장애 시 데이터 손실 가능
- 재생성 가능한 썸네일, 임시 파일에 사용

Glacier Instant Retrieval:
- 분기에 1번 정도 접근하는 아카이브 데이터
- 즉시 조회 가능 (밀리초)
- Standard-IA보다 저렴

Glacier Flexible Retrieval:
- 연 1회 미만 접근
- 조회 시간: 1분 ~ 12시간 선택
- 가격 매우 저렴

Glacier Deep Archive:
- 7년 이상 장기 보관 데이터 (규정 준수)
- 가장 저렴 ($0.00099/GB/월 ≈ Standard의 1/23)
- 조회 시간: 12~48시간
- 예: 금융 거래 내역 7년 보관

Intelligent-Tiering:
- 접근 패턴을 모를 때 사용
- AWS가 자동으로 적절한 티어로 이동
- 자동화 비용 소량 발생
```

## 수명 주기 정책 — 자동 비용 절감

로그 파일 같은 경우 시간이 지날수록 접근 빈도가 줄어든다. 수명 주기 정책으로 자동으로 저렴한 스토리지 클래스로 이동시키거나 삭제할 수 있다.

```json
{
  "Rules": [
    {
      "Id": "logs-lifecycle",
      "Status": "Enabled",
      "Filter": {"Prefix": "logs/"},   // logs/ 폴더에만 적용
      "Transitions": [
        {
          "Days": 30,                   // 30일 후
          "StorageClass": "STANDARD_IA" // Standard-IA로 이동
        },
        {
          "Days": 90,                   // 90일 후
          "StorageClass": "GLACIER"     // Glacier로 이동
        }
      ],
      "Expiration": {
        "Days": 365                     // 1년 후 자동 삭제
      }
    }
  ]
}
```

```bash
# 수명 주기 정책 적용
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-bucket \
  --lifecycle-configuration file://lifecycle.json

# 확인
aws s3api get-bucket-lifecycle-configuration --bucket my-bucket
```

## 버킷 정책 — 접근 제어

버킷 정책으로 누가 버킷에 접근할 수 있는지 세밀하게 제어한다.

```json
// 예시 1: 정적 웹사이트 - 전체 공개 읽기
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "PublicRead",
      "Effect": "Allow",
      "Principal": "*",              // 모든 사람
      "Action": "s3:GetObject",      // 읽기만
      "Resource": "arn:aws:s3:::my-website-bucket/*"
    }
  ]
}
```

```json
// 예시 2: 특정 IAM Role만 허용 (EC2 앱 서버에서만 접근)
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:role/my-app-role"
      },
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::my-app-bucket/*"
    }
  ]
}
```

```bash
# 버킷 정책 적용
aws s3api put-bucket-policy \
  --bucket my-bucket \
  --policy file://bucket-policy.json

# 퍼블릭 접근 차단 설정 (보안 강화)
aws s3api put-public-access-block \
  --bucket my-bucket \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,\
    BlockPublicPolicy=true,RestrictPublicBuckets=true
```

## Presigned URL — 임시 접근 링크

Presigned URL은 **유효 기간이 있는 임시 다운로드/업로드 링크**다. S3 버킷을 비공개로 유지하면서, 특정 파일에 한시적으로 접근을 허용할 때 사용한다.

**사용 예시:**
```
- 사용자에게 프로필 이미지 업로드 URL 제공 (서버를 거치지 않고 S3에 직접 업로드)
- 구매한 사용자에게 다운로드 링크 제공 (1시간 후 만료)
- 내부 직원용 보고서 공유
```

```java
// Spring Boot에서 Presigned URL 생성

@Service
@RequiredArgsConstructor
public class S3Service {

    private final S3Presigner presigner;
    private final S3Client s3Client;

    @Value("${aws.s3.bucket}")
    private String bucket;

    // 업로드용 Presigned URL (클라이언트가 S3에 직접 업로드)
    public String generateUploadUrl(String key, Duration expiry) {
        PutObjectPresignRequest request = PutObjectPresignRequest.builder()
            .signatureDuration(expiry)
            .putObjectRequest(r -> r.bucket(bucket).key(key))
            .build();
        return presigner.presignPutObject(request).url().toString();
    }

    // 다운로드용 Presigned URL
    public String generateDownloadUrl(String key, Duration expiry) {
        GetObjectPresignRequest request = GetObjectPresignRequest.builder()
            .signatureDuration(expiry)
            .getObjectRequest(r -> r.bucket(bucket).key(key))
            .build();
        return presigner.presignGetObject(request).url().toString();
    }

    // 서버에서 직접 S3에 업로드
    public void upload(String key, byte[] data, String contentType) {
        s3Client.putObject(
            PutObjectRequest.builder()
                .bucket(bucket)
                .key(key)
                .contentType(contentType)
                .build(),
            RequestBody.fromBytes(data)
        );
    }

    // 파일 삭제
    public void delete(String key) {
        s3Client.deleteObject(b -> b.bucket(bucket).key(key));
    }
}
```

```java
// S3 설정
@Configuration
public class S3Config {

    @Bean
    public S3Client s3Client() {
        return S3Client.builder()
            .region(Region.AP_NORTHEAST_2)
            .credentialsProvider(DefaultCredentialsProvider.create())  // EC2 Role 자동 사용
            .build();
    }

    @Bean
    public S3Presigner s3Presigner() {
        return S3Presigner.builder()
            .region(Region.AP_NORTHEAST_2)
            .credentialsProvider(DefaultCredentialsProvider.create())
            .build();
    }
}
```

## 정적 웹사이트 호스팅

S3로 HTML/CSS/JS 파일을 서빙할 수 있다. 서버 없이 웹사이트를 운영하는 가장 저렴한 방법이다.

```bash
# 정적 웹사이트 호스팅 활성화
aws s3api put-bucket-website \
  --bucket my-website-bucket \
  --website-configuration '{
    "IndexDocument": {"Suffix": "index.html"},
    "ErrorDocument": {"Key": "404.html"}
  }'

# 퍼블릭 접근 허용 (정적 웹사이트는 필요)
aws s3api delete-public-access-block --bucket my-website-bucket

# 퍼블릭 읽기 정책 적용
aws s3api put-bucket-policy \
  --bucket my-website-bucket \
  --policy '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-website-bucket/*"
    }]
  }'

# 파일 업로드
aws s3 sync ./dist s3://my-website-bucket

# 접속 URL:
# http://my-website-bucket.s3-website.ap-northeast-2.amazonaws.com
# (HTTPS를 원하면 CloudFront + ACM 인증서 사용)
```

## S3 이벤트 트리거

S3에 파일이 업로드될 때 Lambda 함수가 자동으로 실행되도록 설정할 수 있다.

```
활용 예:
- 이미지 업로드 → 자동으로 썸네일 생성
- CSV 업로드 → 자동으로 DB에 데이터 임포트
- 로그 파일 업로드 → 자동으로 분석 파이프라인 실행
```

```bash
# S3 이벤트 알림 설정 (업로드 시 Lambda 실행)
aws s3api put-bucket-notification-configuration \
  --bucket my-bucket \
  --notification-configuration '{
    "LambdaFunctionConfigurations": [
      {
        "LambdaFunctionArn": "arn:aws:lambda:ap-northeast-2:123456789012:function:image-resizer",
        "Events": ["s3:ObjectCreated:*"],
        "Filter": {
          "Key": {
            "FilterRules": [
              {"Name": "prefix", "Value": "uploads/"},  // uploads/ 폴더만
              {"Name": "suffix", "Value": ".jpg"}        // .jpg 파일만
            ]
          }
        }
      }
    ]
  }'
```

## S3 버전 관리

버전 관리를 활성화하면 같은 파일을 덮어써도 이전 버전이 보존된다.

```bash
# 버전 관리 활성화
aws s3api put-bucket-versioning \
  --bucket my-bucket \
  --versioning-configuration Status=Enabled

# 특정 파일의 버전 목록 조회
aws s3api list-object-versions \
  --bucket my-bucket \
  --prefix images/photo.jpg

# 특정 버전으로 복원
aws s3api get-object \
  --bucket my-bucket \
  --key images/photo.jpg \
  --version-id abcXYZ... \
  restored-photo.jpg
```

## 자주 하는 실수

```
1. 버킷을 실수로 공개(Public)로 설정
   - 개인정보, 내부 문서가 인터넷에 공개됨
   - 항상 Block Public Access 설정 먼저 확인
   - AWS는 실수로 공개된 버킷을 알림으로 경고해줌

2. 버킷 이름 중복
   - 전 세계 유일해야 함
   - 회사명+서비스명+환경 조합 권장
   - 예: mycompany-user-uploads-prod

3. 잘못된 리전에 버킷 생성
   - 한국 사용자 서비스라면 ap-northeast-2에 생성
   - EC2와 같은 리전에 두면 데이터 전송 비용 없음

4. 대용량 파일 단순 업로드
   - 100MB 이상은 Multipart Upload 사용
   - aws s3 cp는 자동으로 Multipart 사용
   - SDK에서 직접 구현 시 주의

5. 비용 예상 실수
   - S3 비용 = 저장 비용 + 요청 비용 + 데이터 전송 비용
   - 이미지 수백만 개를 매번 조회하면 요청 비용이 큼
   - CloudFront 앞에 두면 S3 요청 비용 대폭 절감
```
