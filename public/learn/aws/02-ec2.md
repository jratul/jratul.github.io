# EC2 (Elastic Compute Cloud)

EC2는 AWS에서 제공하는 가상 서버다. 클라우드에서 컴퓨터를 임대하는 것과 같다. 물리적 서버를 직접 사고 관리할 필요 없이, 필요할 때 만들고 필요 없으면 꺼두면 된다.

**EC2 = 클라우드 컴퓨터 임대** — 성능도, 운영체제도, 크기도 원하는 대로 고를 수 있다.

---

## 인스턴스 타입 이해하기

EC2는 종류가 수백 가지다. 이름을 보면 특성을 알 수 있다.

```
이름 구조 예시: c6g.xlarge

c       → 패밀리 (용도)
          t = 버스터블 (가성비 좋음, 개발용)
          m = 범용 (CPU와 메모리 균형)
          c = 컴퓨팅 최적화 (CPU 강함)
          r = 메모리 최적화 (RAM 많음)
          i = 스토리지 최적화 (빠른 SSD)
          g = GPU 가속

6       → 세대 (높을수록 최신, 보통 더 저렴하고 성능 좋음)

g       → CPU 아키텍처
          g = Graviton (ARM, AWS 자체 설계, 저렴)
          a = AMD
          i = Intel
          없으면 Intel

.       → 구분자

xlarge  → 크기 (성능/가격 비례)
          nano < micro < small < medium < large < xlarge < 2xlarge < ...
```

### 언제 어떤 타입을 쓸까

```
t4g.micro, t3.small (버스터블):
- 평소에는 저성능, 필요 시 잠깐 고성능 (크레딧 소모)
- 트래픽이 불규칙한 서비스, 개발/테스트 서버
- 가장 저렴 (t4g.micro: 약 $0.0084/시간)
- 주의: 크레딧 소진 시 CPU가 매우 느려짐

m7i, m6i (범용):
- CPU와 메모리가 1:4 비율
- 가장 무난한 선택
- 일반 API 서버, 웹 서버

c7i, c6i (컴퓨팅 최적화):
- CPU가 강함, CPU:메모리 = 1:2
- 게임 서버, 비디오 인코딩, 과학 계산

r7i, r6i (메모리 최적화):
- RAM이 많음, CPU:메모리 = 1:8
- 인메모리 DB, Redis, 빅 데이터 처리

i4i (스토리지 최적화):
- NVMe SSD가 빠름
- 고성능 데이터베이스
```

**초보자 팁:** 처음엔 t3.micro(무료 티어) 또는 t3.small로 시작하자. 나중에 성능이 부족하면 인스턴스 타입을 변경하면 된다.

## EC2 인스턴스 생성의 핵심 요소

인스턴스를 만들 때 결정해야 할 것들이 있다.

```
1. AMI (Amazon Machine Image):
   - OS + 기본 소프트웨어가 포함된 설치 이미지
   - Amazon Linux 2023: AWS에 최적화, 무료
   - Ubuntu 22.04 LTS: 가장 많이 쓰임, 자료 풍부
   - 직접 만든 커스텀 AMI: 기존 설정을 그대로 복사

2. 인스턴스 타입:
   - 위에서 설명한 t3.micro, m5.large 등

3. 키 페어 (SSH 접속용):
   - .pem 파일 = 비밀번호 대신 쓰는 열쇠
   - 분실하면 재발급 불가 → 안전한 곳에 보관!
   - AWS가 공개키 보관, 내가 개인키(.pem) 보관

4. 보안 그룹:
   - 인스턴스에 붙는 방화벽
   - 어떤 포트를, 어디서의 접속을 허용할지

5. 스토리지 (EBS):
   - EC2에 연결되는 가상 하드디스크
   - 기본 8GB (루트 볼륨)
   - 인스턴스 종료 시 삭제 여부 설정 가능
```

## 보안 그룹 설정 — 방화벽 역할

보안 그룹은 **화이트리스트 방식**이다. 허용한 것만 들어오고, 나머지는 자동으로 차단된다.

```
웹 서버용 보안 그룹 예시:

인바운드 규칙 (들어오는 트래픽):
유형      프로토콜  포트   소스
SSH       TCP       22     내 IP (집 IP 또는 회사 IP만!)
HTTP      TCP       80     0.0.0.0/0 (전 세계)
HTTPS     TCP       443    0.0.0.0/0
Custom    TCP       8080   ALB 보안 그룹 (로드밸런서에서만)

DB 서버용 보안 그룹 예시:

인바운드 규칙:
PostgreSQL TCP      5432   앱 서버 보안 그룹 (앱 서버에서만!)
```

**중요한 보안 원칙:**
```
SSH 포트 22:
❌ 소스: 0.0.0.0/0 (절대 안 됨! 전 세계에서 해킹 시도)
✅ 소스: 내 IP만 (내 집 IP, VPN IP)
✅ 소스: Bastion 서버의 보안 그룹

DB 포트:
❌ 소스: 0.0.0.0/0 (인터넷에서 DB 직접 접근? 위험!)
✅ 소스: 앱 서버 보안 그룹만
```

## SSH로 EC2 접속하기

```bash
# 키 파일 권한 설정 (처음 한 번)
# 너무 느슨한 권한이면 SSH가 거부함
chmod 400 my-key.pem

# SSH 접속 (퍼블릭 IP 또는 DNS 사용)
ssh -i my-key.pem ubuntu@ec2-xxx.ap-northeast-2.compute.amazonaws.com
ssh -i my-key.pem ubuntu@1.2.3.4    # 공인 IP로 접속

# Amazon Linux는 ec2-user
ssh -i my-key.pem ec2-user@1.2.3.4
```

```bash
# ~/.ssh/config 파일에 등록하면 짧게 접속 가능
cat >> ~/.ssh/config << 'EOF'
Host myserver
    HostName 1.2.3.4
    User ubuntu
    IdentityFile ~/.ssh/my-key.pem
EOF

# 이후 이렇게 짧게 접속
ssh myserver
```

**접속이 안 될 때 체크리스트:**
```
1. 보안 그룹에 SSH(22번) 허용 됐는지?
2. 내 IP가 소스에 포함됐는지?
3. 키 파일 권한이 400인지? (chmod 400 key.pem)
4. 유저명이 맞는지? (ubuntu vs ec2-user)
5. 퍼블릭 IP가 있는지? (퍼블릭 서브넷에 있는지?)
```

## EC2 기본 운영 명령어

```bash
# 인스턴스 시작/중지/종료
aws ec2 start-instances --instance-ids i-1234567890abcdef0      # 시작
aws ec2 stop-instances --instance-ids i-1234567890abcdef0       # 중지 (저장은 유지)
aws ec2 terminate-instances --instance-ids i-1234567890abcdef0  # 완전 삭제

# 중지 vs 종료 차이:
# stop: 서버 끄기. 데이터 유지. 재시작 가능. EBS 비용은 계속 발생.
# terminate: 완전 삭제. 루트 볼륨도 삭제 (기본). 재시작 불가.

# 인스턴스 정보 확인
aws ec2 describe-instances --instance-ids i-1234567890abcdef0

# EC2 내부에서 자신의 정보 조회 (인스턴스 메타데이터)
curl http://169.254.169.254/latest/meta-data/instance-id       # 인스턴스 ID
curl http://169.254.169.254/latest/meta-data/instance-type     # 인스턴스 타입
curl http://169.254.169.254/latest/meta-data/public-ipv4       # 퍼블릭 IP
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/  # IAM Role 정보
```

## User Data — 인스턴스 시작 시 자동 실행

User Data는 인스턴스가 **처음 시작될 때 딱 한 번** 실행되는 스크립트다. 서버를 만들 때마다 수동으로 설정하는 것이 귀찮을 때 사용한다.

```bash
#!/bin/bash
# EC2 생성 시 자동 실행 스크립트 예시

# 패키지 목록 업데이트
apt update -y

# Java 17 설치
apt install -y openjdk-17-jdk

# 앱 관련 디렉토리 생성
mkdir -p /opt/myapp /var/log/myapp

# S3에서 배포 파일 다운로드
# (EC2에 S3 읽기 권한 IAM Role이 붙어있어야 함)
aws s3 cp s3://my-deploy-bucket/app.jar /opt/myapp/app.jar

# systemd 서비스 등록 (서버 재시작 시 자동 시작)
cat > /etc/systemd/system/myapp.service << 'EOF'
[Unit]
Description=My Spring Boot Application
After=network.target

[Service]
Type=simple
User=ubuntu
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/java -Xmx512m -jar /opt/myapp/app.jar
Restart=always
RestartSec=10
Environment=SPRING_PROFILES_ACTIVE=prod

[Install]
WantedBy=multi-user.target
EOF

# 서비스 활성화 및 시작
systemctl daemon-reload
systemctl enable myapp
systemctl start myapp
```

## Elastic IP — 고정 IP 주소

EC2를 중지했다가 다시 시작하면 퍼블릭 IP가 바뀐다. Elastic IP를 사용하면 고정 IP를 유지할 수 있다.

```bash
# 고정 IP 할당
aws ec2 allocate-address --domain vpc
# → AllocationId: eipalloc-xxx

# 인스턴스에 연결
aws ec2 associate-address \
  --instance-id i-1234567890 \
  --allocation-id eipalloc-xxx

# 인스턴스에서 분리
aws ec2 disassociate-address --association-id eipassoc-xxx

# 반환 (더 이상 필요 없을 때)
aws ec2 release-address --allocation-id eipalloc-xxx
```

**Elastic IP 과금 주의:**
```
Elastic IP가 인스턴스에 연결되어 있으면: 무료
Elastic IP를 할당했지만 연결하지 않으면: 유료 ($0.005/시간 = 월 약 $3.6)

→ 사용하지 않는 Elastic IP는 즉시 반환하자!
```

## Auto Scaling — 자동으로 서버 수 조절

Auto Scaling은 트래픽에 따라 EC2 인스턴스 수를 자동으로 늘리고 줄인다. 낮에는 10대, 밤에는 2대 식으로 자동 운영 가능하다.

```
Auto Scaling Group (ASG) 구성:

Launch Template    → 어떤 인스턴스를 만들지 정의
  - AMI, 인스턴스 타입, 보안 그룹, User Data 등

Auto Scaling Group → 인스턴스 수 관리
  - 최소: 2 (항상 최소 2대 유지)
  - 최대: 10 (최대 10대까지만)
  - 원하는 수: 2 (평상시 2대)

스케일링 정책:
  Target Tracking  → CPU 70% 유지되도록 자동 조절 (권장)
  Step Scaling     → CPU 80% 넘으면 2대 추가, 90% 넘으면 4대 추가
  Scheduled        → 매일 오전 9시에 5대, 오후 9시에 2대
```

```bash
# Launch Template 생성
aws ec2 create-launch-template \
  --launch-template-name my-app-lt \
  --version-description "v1" \
  --launch-template-data '{
    "ImageId": "ami-xxx",
    "InstanceType": "t3.small",
    "KeyName": "my-key",
    "SecurityGroupIds": ["sg-xxx"],
    "IamInstanceProfile": {"Name": "my-instance-profile"}
  }'

# Auto Scaling Group 생성
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name my-app-asg \
  --launch-template LaunchTemplateName=my-app-lt,Version='$Latest' \
  --min-size 2 \             # 최소 2대
  --max-size 10 \            # 최대 10대
  --desired-capacity 2 \     # 처음엔 2대
  --vpc-zone-identifier "subnet-xxx,subnet-yyy" \  # 여러 AZ에 분산
  --target-group-arns arn:aws:elasticloadbalancing:...  # ALB 연결
```

## 인스턴스 구매 전략 — 비용 절감

```
실전 조합 (30-70% 비용 절감):

베이스 트래픽 처리:
→ Reserved Instance 2대 (1년 약정, 40% 할인)
→ 항상 실행되는 최소 서버

추가 트래픽 처리:
→ Spot Instance로 Auto Scaling
→ Spot 중단 시 On-Demand로 자동 전환

예시:
API 서버: Reserved 2대 (기본) + Spot 최대 8대 (피크)
배치 서버: 100% Spot (중단되어도 재시작하면 됨)
```

## EC2 모니터링 기본

```bash
# CloudWatch 기본 지표 (EC2 콘솔에서도 확인 가능)
- CPUUtilization       → CPU 사용률 (%)
- NetworkIn/Out        → 네트워크 입출력 (bytes)
- DiskReadOps/WriteOps → 디스크 I/O
- StatusCheckFailed    → 인스턴스 상태 이상

# 메모리 사용률은 기본 지표에 없음!
# CloudWatch Agent 설치 필요
sudo yum install -y amazon-cloudwatch-agent  # Amazon Linux
sudo apt install -y amazon-cloudwatch-agent  # Ubuntu

# 상태 확인
aws ec2 describe-instance-status --instance-ids i-xxx
```

## 자주 하는 실수

```
1. 키 페어 분실
   - .pem 파일을 잃어버리면 그 인스턴스는 SSH 접속 불가
   - 백업해두거나 AWS Systems Manager Session Manager 사용

2. 인스턴스 종료 후 데이터 손실
   - terminate 시 루트 볼륨 자동 삭제 (기본 설정)
   - 중요한 데이터는 EBS를 별도로 붙이거나 S3에 백업

3. 퍼블릭 서브넷에 DB 서버 배치
   - DB는 반드시 프라이빗 서브넷에!

4. 보안 그룹 SSH를 0.0.0.0/0으로 설정
   - 전 세계에서 무차별 대입 공격이 들어옴

5. 비용 알림 없이 테스트하다 큰 요금 청구
   - AWS Budgets 먼저 설정하고 시작!
```
